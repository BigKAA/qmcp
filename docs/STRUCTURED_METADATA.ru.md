# Структурированные метаданные для фильтрации

> **Документация:** [English](./STRUCTURED_METADATA.md) | Русский

## Обзор

Этот документ описывает реализацию структурированных метаданных для Qdrant MCP Server, позволяющих выполнять расширенную фильтрацию при семантическом поиске.

### Что такое структурированные метаданные?

Структурированные метаданные — это обогащённая информация, извлекаемая из исходного кода при индексации. Она предоставляет детальные сведения о каждом фрагменте кода и позволяет выполнять точную фильтрацию для более релевантных результатов поиска.

### Зачем нужны структурированные метаданные?

Без фильтрации по метаданным семантический поиск может возвращать нерелевантные результаты. Например, поиск "аутентификация" может вернуть документацию об auth в совершенно другом контексте. Со структурированными метаданными:

- Находите точные определения символов по имени
- Фильтруйте результаты по типу кода (функции, классы, файлы)
- Ищите в пределах определённых языков программирования
- Комбинируйте фильтры для точного поиска

## Новые поля в Qdrant Payload

Каждый проиндексированный фрагмент теперь содержит следующие расширенные метаданные:

```json
{
    "file_path": "src/auth/users.py",
    "content": "def get_user(user_id: int) -> User:\n    \"\"\"Получить пользователя по ID.\"\"\"\n    session = Session()\n    return session.query(User).get(user_id)",
    "chunk_type": "function_def",
    "signature": "def get_user(user_id: int) -> User",
    "symbol_names": ["get_user", "user_id", "User", "session", "Session", "query"],
    "imports": ["from models import User", "from db import Session"],
    "language": "python"
}
```

### Описание полей

| Поле | Тип | Описание |
|------|-----|---------|
| `file_path` | `str` | Относительный путь к исходному файлу |
| `content` | `str` | Содержимое фрагмента кода |
| `chunk_type` | `str` | Классификация фрагмента кода |
| `signature` | `str \| None` | Полная сигнатура функции/класса (только Python) |
| `symbol_names` | `list[str]` | Все идентификаторы найденные в фрагменте |
| `imports` | `list[str]` | Операторы импорта из исходного файла |
| `language` | `str` | Идентификатор языка программирования |

## Параметры фильтрации в qdrant_search

Инструмент `qdrant_search` теперь принимает следующие параметры фильтрации:

### Доступные фильтры

| Параметр | Тип | Описание | Примеры значений |
|----------|-----|---------|-----------------|
| `chunk_type` | `str \| None` | Тип фрагмента кода для фильтрации | `"function_def"`, `"class_def"` |
| `symbol_name` | `str \| None` | Точное совпадение имени символа | `"get_user"`, `"User"` |
| `language` | `str \| None` | Язык программирования | `"python"`, `"go"`, `"typescript"` |

### Сигнатура API

```python
@mcp.tool()
async def qdrant_search(
    query: str,
    collection: str,
    limit: int = 5,
    score_threshold: float = 0.7,
    chunk_type: str | None = None,      # Фильтр по типу фрагмента
    symbol_name: str | None = None,      # Точное совпадение символа
    language: str | None = None,          # Фильтр по языку
) -> list[dict[str, Any]]:
    """
    Семантический поиск с опциональными фильтрами метаданных.
    
    Все параметры фильтрации опциональны. Если они не указаны,
    поиск работает как раньше (без фильтрации).
    """
```

## Подробные примеры использования

### Пример 1: Найти функции по точному имени

**Сценарий:** Вам нужно найти где определена конкретная функция во всей кодовой базе.

```
Запрос пользователя: "Найди где определена функция get_user"
```

**Вызов API:**
```python
results = await qdrant_search(
    query="user authentication get_user",
    collection="myproject",
    symbol_name="get_user",
    limit=10
)
```

**Результат:** Возвращает только фрагменты кода где `get_user` фигурирует как определённый символ, ранжированные по семантическому сходству с запросом.

**Почему это работает:** Поле `symbol_names` содержит все идентификаторы, извлечённые из каждого фрагмента. При указании `symbol_name="get_user"` механизм фильтрации Qdrant гарантирует возврат только тех фрагментов, которые содержат этот символ.

---

### Пример 2: Искать только определения классов

**Сценарий:** Вы хотите найти все модели данных и структуры классов в вашем проекте.

```
Запрос пользователя: "модели данных и структуры"
```

**Вызов API:**
```python
results = await qdrant_search(
    query="data model structure schema",
    collection="myproject",
    chunk_type="class_def",
    limit=10
)
```

**Результат:** Возвращаются только Python определения классов `class`, отфильтрованные из полных семантических результатов.

**Почему это работает:** Фильтр `chunk_type="class_def"` ограничивает результаты фрагментами, классифицированными как определения классов. Другой код (функции, импорты и т.д.) исключается.

---

### Пример 3: Фильтрация по языку программирования

**Сценарий:** Вы хотите найти реализации API эндпоинтов, но только в Python файлах.

```
Запрос пользователя: "API эндпоинты"
```

**Вызов API:**
```python
results = await qdrant_search(
    query="api endpoint route handler",
    collection="myproject",
    language="python",
    limit=10
)
```

**Результат:** Возвращает реализации API эндпоинтов на Python, исключая Go, TypeScript или реализации на других языках.

**Почему это работает:** Поле `language` устанавливается при парсинге на основе расширения файла. Фильтрация по `"python"` гарантирует включение только исходных файлов Python.

---

### Пример 4: Комбинирование нескольких фильтров

**Сценарий:** Вам нужно найти определения функций связанных с аутентификацией только на Python.

```
Запрос пользователя: "логика аутентификации"
```

**Вызов API:**
```python
results = await qdrant_search(
    query="authentication login password verify",
    collection="myproject",
    chunk_type="function_def",
    language="python",
    limit=10
)
```

**Результат:** Возвращает только определения функций Python соответствующие семантическому запросу об аутентификации.

**Почему это работает:** Несколько фильтров комбинируются с помощью AND логики. Фрагмент должен удовлетворять ВСЕМ указанным фильтрам чтобы попасть в результаты.

---

### Пример 5: Обнаружение асинхронных функций

**Сценарий:** Поиск всех асинхронных операций в Python кодовой базе.

```
Запрос пользователя: "асинхронные операции"
```

**Вызов API:**
```python
results = await qdrant_search(
    query="async await fetch stream",
    collection="myproject",
    chunk_type="async_function_def",
    limit=10
)
```

**Результат:** Возвращаются только определения функций `async def`.

### Пример 6: Поиск по перекрёстным ссылкам

**Сценарий:** Вы хотите найти файлы которые импортируют конкретный модуль.

## Детали реализации

### Dataclass ParsedChunk

Основная структура данных для проиндексированных фрагментов кода:

```python
# src/qmcp/parser/base.py
from dataclasses import dataclass, field
from typing import Optional

@dataclass
class ParsedChunk:
    """
    Представляет распарсенный фрагмент кода с извлечёнными метаданными.
    
    Атрибуты:
        file_path: Относительный путь к исходному файлу
        content: Содержимое фрагмента кода
        chunk_type: Классификация (function_def, class_def, file и т.д.)
        name: Имя основного символа (имя функции, класса и т.д.)
        line_start: Начальный номер строки в исходном файле
        line_end: Конечный номер строки в исходном файле
        docstring: Текст присоединённого docstring если есть
        signature: Полная сигнатура функции/класса (на основе AST для Python)
        symbol_names: Все идентификаторы найденные в фрагменте
        imports: Операторы импорта из исходного файла
        language: Идентификатор языка программирования
    """
    file_path: str
    content: str
    chunk_type: str
    name: str
    line_start: int
    line_end: int
    docstring: str = ""
    signature: Optional[str] = None       # НОВОЕ: "def foo(a: int) -> str"
    symbol_names: list[str] = field(default_factory=list)  # НОВОЕ: ["foo", "a"]
    imports: list[str] = field(default_factory=list)       # НОВОЕ: ["import os"]
    language: str = "unknown"             # НОВОЕ: "python"
```

### Реализация Python парсера

Python парсер (`src/qmcp/parser/python.py`) извлекает метаданные с помощью модуля AST Python:

#### Извлечение сигнатуры

```python
def _extract_signature(self, node: ast.FunctionDef | ast.AsyncFunctionDef | ast.ClassDef) -> str:
    """
    Реконструирует сигнатуру функции/класса из AST узла.
    
    Возвращает строку вида: "def function_name(arg1: Type1, arg2: Type2) -> ReturnType"
    
    Обрабатывает:
    - Позиционные и именованные аргументы
    - Значения по умолчанию
    - Аннотации типов
    - *args и **kwargs
    """
```

#### Извлечение имён символов

```python
def _extract_symbol_names(self, node: ast.AST) -> list[str]:
    """
    Извлекает все идентификаторы из AST узла.
    
    Включает:
    - Имя самой функции/класса
    - Имена параметров
    - Ссылки на типы аннотаций
    - Присваивания локальных переменных в теле
    - Имена вызываемых функций
    """
```

#### Извлечение импортов

```python
def _extract_imports(self, tree: ast.AST) -> list[str]:
    """
    Извлекает все операторы импорта из файла.
    
    Возвращает список строк вида:
    - "import os"
    - "from typing import Optional, List"
    - "from models import User, Session"
    """
```

## Справочник типов фрагментов

| chunk_type | Описание | Узел AST | Пример |
|------------|----------|----------|--------|
| `function_def` | Определение функции Python | `ast.FunctionDef` | `def hello():` |
| `async_function_def` | Асинхронная функция Python | `ast.AsyncFunctionDef` | `async def fetch():` |
| `class_def` | Определение класса Python | `ast.ClassDef` | `class User:` |
| `file` | Фрагмент многоязычного файла | N/A (tree-sitter) | Файлы Go, JS, TS |
| `document` | Файл документации | N/A | Markdown файлы |

## Обратная совместимость

### Принципы проектирования

Все изменения полностью совместимы с предыдущими версиями:

1. **Опциональные фильтры:** Все новые параметры фильтрации опциональны (`None` по умолчанию). Когда они не указаны, поиск работает точно так же как раньше.

2. **Nullable поля:** Новые поля payload (`signature`, `symbol_names`, `imports`, `language`) могут быть null. Более старые проиндексированные данные без этих полей будут иметь `None` или пустые значения.

3. **Плавная деградация:** Поиск работает с частичными метаданными:
   - Фрагменты без `signature` всё ещё ищутся по содержимому
   - Фрагменты без `symbol_names` всё ещё находятся через фильтр `chunk_type`
   - Фрагменты без `language` всё ещё ищутся без фильтра по языку

### Путь миграции

Существующие проиндексированные данные не требуют переиндексации. Однако для использования новых возможностей фильтрации:

```python
# Старые данные работают без изменений
results = await qdrant_search(
    query="authentication",
    collection="myproject"
)  # Работает точно так же как раньше

# Новая фильтрация доступна когда нужно
results = await qdrant_search(
    query="authentication",
    collection="myproject",
    chunk_type="function_def"  # Теперь доступно
)
```

Пotentialные будущие возможности:

## Краткий справочник

### Поиск с фильтрами

```python
# Найти конкретную функцию
qdrant_search(query="...", symbol_name="get_user")

# Найти только классы  
qdrant_search(query="...", chunk_type="class_def")

# Найти только код на Python
qdrant_search(query="...", language="python")

# Комбинировать фильтры
qdrant_search(
    query="authentication",
    chunk_type="function_def",
    language="python"
)
```
